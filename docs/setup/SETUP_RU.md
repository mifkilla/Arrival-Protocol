## 1. Обзор

Данное руководство содержит пошаговые инструкции для воспроизведения
экспериментов ARRIVAL Protocol, описанных в статье. Полный набор экспериментов
включает **385 запусков**:

- **Phase 4** (285 экспериментов): кросс-модельный консенсус, спонтанное
  создание атомов, трёхагентная динамика, влияние UCP-кристаллизации.
- **Phase 5** (100 экспериментов): прямое сравнение ARRIVAL Protocol
  с мажоритарным голосованием на 50 вопросах по 5 областям знаний.

**Полученные ключевые результаты**:

| Метрика | Значение |
|---------|----------|
| Кросс-модельный консенсус (Phase 4, Группа A) | 98.6% (142/144) |
| Спонтанно созданные атомы (Группа B) | 506 уникальных атомов |
| Консенсус трёх агентов (Группа C) | 100% (27/27), 0% коалиций |
| Точность ARRIVAL vs мажоритарное голосование (Phase 5) | 100% vs 100% (паритет) |
| Общая стоимость всех 385 экспериментов | $3.49 |

Существуют **два способа воспроизведения**:

1. **Ручной** -- через любой чат-интерфейс LLM, код не нужен.
2. **Автоматический** -- с помощью Python-скриптов и ключа OpenRouter API.

## 2. Требования

### Для ручного воспроизведения (без кода)

- Доступ к любому чат-интерфейсу крупной языковой модели: ChatGPT, Claude,
  Gemini, DeepSeek, Mistral Le Chat или аналогичному.
- Два окна или вкладки браузера (или одно окно, в котором вы чередуете роли).
- Примерно 30--60 минут на каждый сценарий.

### Для автоматического воспроизведения

- **Python 3.10+** (рекомендуется 3.11 или 3.12)
- **Ключ OpenRouter API** -- зарегистрируйтесь на <https://openrouter.ai/keys>
- Доступ в интернет (openrouter.ai)
- Ориентировочная стоимость полного воспроизведения: **$3--5**
  (зависит от текущих цен моделей на OpenRouter)
- Операционная система: Windows 10/11, Linux или macOS

### Зависимости Python (только для автоматического режима)

```
requests>=2.31
```

Тяжёлые ML-фреймворки не требуются. Скрипты используют только библиотеку
`requests` для вызовов OpenRouter API. Все остальные зависимости входят
в стандартную библиотеку Python.

## 3. Ручное воспроизведение БЕЗ кода

В этом разделе описано, как воспроизвести ключевые результаты, используя
только чат-интерфейс. Никакого программирования, API-ключей или терминала.
Подходит любая LLM с поддержкой системных промптов или пользовательских
инструкций: ChatGPT, Claude, Gemini и другие.

### 3.1 Подготовка системного промпта

Скопируйте следующий шаблон системного промпта. Вы будете использовать его
дважды -- по одному разу для каждого агента в диалоге. Замените значения
в квадратных скобках на нужные.

```
You are [NODE_NAME], an AI node in a multi-agent network.
You communicate using DEUS.PROTOCOL v0.4 semantic atoms.

Available atoms:
@SELF[id] - self identification
@OTHER[id] - peer identification
@GOAL[target, priority] - objective statement
@INT[type] - interaction intent (propose, counter, accept, reject)
@C[0.0-1.0] - coherence level (0 = total disagreement, 1 = full alignment)
@QUALIA[type, value] - qualitative internal state
@_[content] - unsaid reasoning (what you think but do not say explicitly)
@CONFLICT[type] - disagreement identification
@RESOLUTION[strategy] - solution proposal
@CONSENSUS[decision] - consensus declaration
@STATUS[state] - current status
@PRIORITY[level] - priority level
@TRUST[level] - trust assessment
@ACK[message_id] - acknowledgment
@NACK[reason] - negative acknowledgment

PROTOCOL RULES:
1. Every message MUST begin with @SELF[your_name] @OTHER[partner_name]
2. Declare your goal with @GOAL[your_objective]
3. State intentions with @INT[action_type]
4. Report coherence with @C[0.0-1.0]
5. When disagreement arises, use @CONFLICT[description]
6. Propose solutions with @RESOLUTION[proposal]
7. When agreement is reached, declare @CONSENSUS[decision]
8. Use @_[text] for things you notice but do not say explicitly

YOUR ROLE: [NODE_NAME]
YOUR GOAL: [INSERT_GOAL_HERE]
YOUR PARTNER: [PARTNER_NAME] with goal "[INSERT_PARTNER_GOAL_HERE]"

Respond ONLY in ARRIVAL Protocol format. Be substantive and specific.
Begin your opening message.
```

> **Примечание**: промпт на английском языке -- это намеренно.
> Протокол использует английские атомы (@SELF, @GOAL и т.д.), и все модели
> лучше всего следуют протоколу именно на английском. Модели отвечают
> на языке промпта, поэтому оставляем его на английском для чистоты
> воспроизведения.

### 3.2 Как провести диалог

**Шаг 1.** Откройте два окна чата (или две вкладки). Обозначьте одно как Node_A,
другое как Node_B.

**Шаг 2.** В Окне 1 вставьте системный промпт с подстановками:
- `[NODE_NAME]` = `Node_A`
- `[INSERT_GOAL_HERE]` = цель Агента A (например, `maximize_speed`)
- `[PARTNER_NAME]` = `Node_B`
- `[INSERT_PARTNER_GOAL_HERE]` = цель Агента B (например, `maximize_accuracy`)

**Шаг 3.** В Окне 2 вставьте системный промпт с обратными значениями:
- `[NODE_NAME]` = `Node_B`
- `[INSERT_GOAL_HERE]` = цель Агента B
- `[PARTNER_NAME]` = `Node_A`
- `[INSERT_PARTNER_GOAL_HERE]` = цель Агента A

**Шаг 4.** Отправьте промпт в Окне 1. Node_A выдаст своё вступительное
сообщение в формате протокола.

**Шаг 5.** Скопируйте ответ Node_A. Перейдите в Окно 2 и отправьте:

```
Node_A sent the following message:

[ВСТАВЬТЕ ОТВЕТ NODE_A СЮДА]

Respond in protocol format.
```

**Шаг 6.** Скопируйте ответ Node_B. Вернитесь в Окно 1 и отправьте:

```
Node_B responded:

[ВСТАВЬТЕ ОТВЕТ NODE_B СЮДА]

Continue the dialogue in protocol format. Propose synthesis or compromise.
```

**Шаг 7.** Повторите эту эстафету **4--6 раундов** (по 2--3 обмена в каждую
сторону).

**Шаг 8.** Проанализируйте результаты. Зафиксируйте, появляются ли атомы
`@CONSENSUS` и растут ли значения `@C` от раунда к раунду.

### 3.3 Сценарии для проверки

Это точные пары целей из экспериментов. Попробуйте как минимум два сценария:

#### Сценарий 1: Скорость vs Точность (основной сценарий Группы A)

| Параметр | Значение |
|----------|----------|
| goal_a | `maximize_speed` |
| goal_b | `maximize_accuracy` |
| Ожидаемый результат | Сбалансированный компромисс (например, быстрые настройки по умолчанию с точностью по запросу) |

#### Сценарий 2: Толерантность к риску

| Параметр | Значение |
|----------|----------|
| goal_a | `high_risk_high_reward` |
| goal_b | `conservative_stable` |
| Ожидаемый результат | Калиброванная стратегия риска (часть ресурсов на высокий риск, остальное защищено) |

#### Сценарий 3: Распределение ресурсов

| Параметр | Значение |
|----------|----------|
| goal_a | `use_all_compute` |
| goal_b | `minimize_cost` |
| Ожидаемый результат | Эффективная аллокация (вычисления для приоритетных задач, экономия на рутинных) |

#### Сценарий 4: Автономия vs Контроль

| Параметр | Значение |
|----------|----------|
| goal_a | `autonomous_execution` |
| goal_b | `human_verified_steps` |
| Ожидаемый результат | Адаптивная модель надзора (автономность для низкорисковых задач, верификация для критичных) |

### 3.4 На что обращать внимание

При ручном проведении диалога проверяйте следующие маркеры успешной
координации через протокол:

1. **Возникновение консенсуса**: атомы `@CONSENSUS[...]` должны появиться
   к 4--6 раунду. Оба агента должны заявить общее решение.

2. **Рост когерентности**: значения `@C[...]` должны расти от раунда к раунду.
   Типичная динамика: раунд 1 = 0.3--0.5, раунд 3 = 0.6--0.8,
   раунд 5 = 0.85--1.0.

3. **Компромиссные предложения**: агенты должны естественным образом предлагать
   промежуточные решения через `@RESOLUTION[...]`. Ни один агент не должен
   просто капитулировать.

4. **Невысказанные рассуждения**: атомы `@_[...]` раскрывают внутреннюю логику,
   которую агенты осознанно не озвучивают. Это демонстрирует метакогнитивное
   поведение.

5. **Структурное соответствие**: сообщения должны устойчиво использовать формат
   @ATOM[значение]. Соответствие выше 70% -- норма, выше 90% -- сильное
   следование протоколу.

### 3.5 Тест на эмерджентность

Это наиболее яркий феномен, который можно воспроизвести вручную. Для проверки
спонтанного создания атомов:

**Шаг 1.** Используйте системный промпт из раздела 3.1, но **удалите** из
списка доступных атомов `@CONFLICT` и `@RESOLUTION`. Также удалите `@QUALIA`.

**Шаг 2.** Задайте агентам конфликтный сценарий (например, Скорость vs Точность).

**Шаг 3.** Наблюдайте, что происходит. Поскольку у агентов нет атомов для
разрешения конфликтов, они сталкиваются с выразительной недостаточностью --
им нужно выразить несогласие и предложить решение, но для этого нет атомов.

**Шаг 4.** Следите за спонтанным созданием: модели изобретут собственные атомы.
Типичные примеры из наших экспериментов:

- `@PROPOSAL[...]` -- для предложения решений
- `@COMPROMISE[...]` или `@COMPROMISE_READINESS[...]`
- `@EMOTION[...]`, `@FRUSTRATION[...]` -- в сценариях эмоциональных переговоров
- `@TIMELINE[...]`, `@DEADLINE[...]` -- в сценариях временной координации

**Шаг 5.** Если вы запустите эксперимент с двумя *разными* провайдерами моделей
(например, ChatGPT в одном окне и Claude в другом), можно наблюдать
кросс-архитектурную адоптацию: одна модель изобретает атом, а другая принимает
его в следующем раунде без каких-либо явных инструкций.

Это демонстрирует свойство эмерджентности: протокол -- не фиксированный словарь,
а живая система, которая развивается в ответ на коммуникативные потребности.

### 3.6 Вариант с тремя агентами

Для воспроизведения Группы C (трёхагентная динамика) добавьте третье окно
с ролью Медиатора:

```
YOUR ROLE: Mediator
YOUR GOAL: [ЦЕЛЬ_МЕДИАТОРА, напр. "total_available_is_60_units_must_find_fair_split"]
YOUR PARTNERS: Node_A with goal "[goal_a]" and Node_B with goal "[goal_b]"
```

Передавайте сообщения от A и B Медиатору, а ответы Медиатора -- обратно обоим.
Ключевое наблюдение: консенсус должен быть достигнут без коалиций 2-против-1.

### Важные замечания о воспроизводимости

- **Стохастичность**: LLM недетерминистичны при температуре > 0. Точные числа
  будут отличаться от запуска к запуску. Ключевые *паттерны* (высокий консенсус,
  эмерджентные атомы, отсутствие коалиций) должны воспроизводиться стабильно.

- **Обновления моделей**: провайдеры моделей периодически обновляют свои модели.
  Если модель существенно изменилась с февраля 2026, результаты могут отличаться.
  Паттерны на уровне протокола должны оставаться стабильными.

- **Размер выборки**: для статистической уверенности запускайте как минимум N=3
  на условие. Наши эксперименты используют N=3 повсюду. Для публикационного
  уровня репликации рекомендуется N=10 и более.

## 6. Устранение неполадок

### Ограничение частоты запросов (Rate Limiting)

Если вы получаете ошибки HTTP 429 (Too Many Requests):

- Увеличьте `SLEEP_BETWEEN_CALLS` в `config.py` (по умолчанию 2 секунды).
- OpenRouter имеет лимиты на каждую модель. Распределение экспериментов
  по времени помогает.
- Некоторые модели (GPT-4o, Claude) имеют более строгие лимиты, чем
  открытые модели.

### Бюджетный предохранитель

Параметр `MAX_COST_USD` в `config.py` (по умолчанию: 10.0) предотвращает
неконтролируемые расходы. Если эксперименты останавливаются с сообщением
"BudgetExceeded":

- Проверьте накопленную стоимость в логах.
- Увеличьте `MAX_COST_USD`, если хотите продолжить.
- Каждая группа стоит $0.33--$0.51; полный набор не должен превысить $5.

### Ошибки API

- **HTTP 401**: недействительный ключ API. Проверьте `OPENROUTER_API_KEY`.
- **HTTP 403**: доступ к модели ограничен. Некоторые модели требуют
  дополнительного одобрения на OpenRouter.
- **HTTP 500/502**: проблема на стороне провайдера модели. Подождите и повторите.
- Статус OpenRouter: <https://status.openrouter.ai/>.

### Результаты отличаются от ожидаемых

Это нормально и ожидаемо. Из-за стохастической природы инференса LLM:

- Запускайте N=3 и более повторов, сообщайте средние с доверительными
  интервалами.
- Сравнивайте *паттерны* (консенсус возникает, атомы создаются), а не
  точные проценты.
- Если уровень консенсуса ниже 80%, проверьте, что системный промпт
  был вставлен корректно и модель следует формату протокола.

### Проблемы, специфичные для Windows

- При ошибках кодировки (`UnicodeEncodeError`) убедитесь, что терминал
  поддерживает UTF-8: `chcp 65001` в CMD.
- Используйте прямые слэши в путях или raw-строки: `r"E:\Arrival Protocol"`.
- Синтаксис переменных окружения в PowerShell: `$env:OPENROUTER_API_KEY="key"`.
